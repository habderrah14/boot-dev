# Chapter 03 — GitHub Profile

> *"A recruiter has 30 seconds on your GitHub. Half of them won't scroll past the fold. The top of your profile is a billboard — treat it like one."*

## Learning objectives

By the end of this chapter you will be able to:

- Configure your GitHub profile so the top 500 pixels signal exactly what you want.
- Pin the right repositories with titles and descriptions that hiring managers can skim.
- Clean up repo noise — abandoned forks, tutorial copies, and half-finished experiments.
- Use the profile README to bridge the gap between "code" and "hire me."
- Maintain your profile as a living document, not a one-time setup.

## The simple version

When someone opens `github.com/yourname`, they see — from top to bottom — your avatar, name, bio, a profile README (if you've set one up), six pinned repos, and the contribution graph. The recruiter might never scroll past the pinned repos. Your job is to engineer those top elements so they tell a coherent story: who you are, what you build, and why someone should click deeper.

The other half of this work is noise reduction. If you have 25 public repos and 20 of them are tutorial clones, half-started experiments, and untouched forks, the signal-to-noise ratio works against you. Archive or hide the noise; pin the signal.

## Concept deep-dive

### Username, avatar, name, bio

Quick checks before anything else:

- **Username.** Pronounceable, professional, ideally matches your other handles (`firstname-lastname`, `flastname`). If you're stuck with a gaming handle from 2014, create a new account.
- **Avatar.** A photo of your face or a clean graphic. Not the default octopus — it signals "just signed up."
- **Display name.** Your real name. Recruiters Google you; make the match easy.
- **Bio.** One sentence: who you are + what you do. "Backend engineer — Python, TypeScript, Postgres. Building habits, not streaks." Avoid buzzword soup.
- **Contact.** Link to LinkedIn + portfolio + email. At least two of three.

### The profile README

Create a repo named *exactly* like your username. Its `README.md` renders on your profile page. Minimum viable version:

```markdown
# Hi, I'm [Name]

Backend engineer focused on [area].
Currently [learning / building / looking for junior roles].

- Strong in: Python, TypeScript, SQL, Linux.
- Interested in: distributed systems, developer tooling, fintech.
- Portfolio: [yourdomain.com](https://yourdomain.com)
- Reach me: [linkedin.com/in/you](https://linkedin.com/in/you)

## What I'm building

- [goals-api](https://github.com/you/goals-api) — deployed
  habit-tracking API with Postgres + OAuth.
- [tiny-queue](https://github.com/you/tiny-queue) — a small job
  queue written to understand RabbitMQ.

## What I'm learning right now

Kafka and event-driven systems.
```

Avoid: autogenerated stats cards (noisy, say nothing specific), animated snakes eating contributions (reviewers notice — not positively), and walls of emoji headers.

### Pinned repos: the six slots

You get six pins. Treat them as the six most valuable artifacts on your entire profile. Order matters — top-left is read first.

A good ordering for a junior backend candidate:

1. **Headline project** (the main portfolio app).
2. **Depth project** (the system-y piece).
3. **Collaboration** (a fork with a landed PR, or a shared repo).
4. **Focused utility or library** you built.
5. **Course/book work repo** — only if it has serious notes or diagrams.
6. **Writeup repo** — a README-only repo with a technical deep-dive.

If you don't have six strong candidates, **pin fewer**. Four strong pins beat six mixed ones.

### Repository-level polish

For every pinned repo, verify:

- **Description** (repo Settings → About). One clear sentence.
- **Topics** (tags). 3–5 relevant ones. Boosts GitHub search.
- **Homepage URL** — the live demo, if deployed.
- **README** — the detailed content (chapter 02's structure).
- **License file** — open-source convention; shows professionalism.
- **Recent commit** — an active repo signals life. A minor commit every month or two is fine.

### Noise reduction

Go through *every* public repo and pick one action:

- **Keep** (pinned or supporting).
- **Archive** (read-only, grayed out, still visible). Good for honest old work.
- **Make private** (removes from profile entirely).
- **Delete** (if it's offensive, broken, or embarrassing).

Typical cleanup: 25 public repos → 8 public, 4 pinned. Signal density goes up; "tutorial graveyard" impression goes away.

### Forks are usually noise

A fork sitting untouched on your profile signals "I clicked the fork button." Either make a visible contribution and leave it (link the merged PR in the README), or delete the fork. The contribution graph credits your work regardless.

### The contribution graph

A sparse graph is fine. An empty graph for the last 3 months while you claim to be job-hunting looks strange. Gentle daily activity — a commit to notes, a PR review, small code changes — looks healthy. Don't *manufacture* commits with bots — engineers can tell, and it reads as desperate.

### Naming your projects

Project names are marketing. Neutral, descriptive names age better than clever ones:

- Good: `goals-api`, `tiny-queue`, `markdown-fs`
- Less good: `app2`, `final-project-2`, `nodejs-project`
- Bad: `dark-raven-engine-69`, `my-sick-api`, `untitled-3`

## Why these design choices

**Why so much emphasis on the "above the fold"?** Because recruiters and engineers have dozens of candidates to review. The ones who make the first cut are the ones whose profiles communicate clearly in 15 seconds. Below-the-fold content is a bonus, not the pitch.

**Why pin fewer than six if you don't have six strong repos?** Because filler repos dilute the signal. Three strong pins followed by three mediocre ones leave the reviewer with a weaker impression than three strong pins followed by nothing.

**Why archive instead of delete old projects?** Because archived repos show an honest history — you tried things, learned, and moved on. Deleting everything except your best work can look artificially polished. Archive the middling stuff; delete the genuinely embarrassing stuff.

**When you'd pick differently:** If you're applying to a very specific niche (e.g., open-source maintainer role), curate your profile heavily around that niche — even if it means hiding strong work that's off-topic.

## Production-quality code

### Profile README template

```markdown
# Hi, I'm [Your Name]

[One sentence: role + focus + current status.]

- **Strong in:** [3-5 technologies you'd defend in an interview]
- **Interested in:** [2-3 areas that match your target role]
- **Portfolio:** [URL]
- **Contact:** [LinkedIn URL] · [email]

## Featured projects

- **[Project 1](link)** — [one-line pitch]
- **[Project 2](link)** — [one-line pitch]
- **[Project 3](link)** — [one-line pitch]

## Currently learning

[One sentence about what you're studying right now.]
```

### Profile audit checklist

```
[ ] Username is professional and pronounceable
[ ] Avatar is a real photo or clean graphic (not default)
[ ] Display name matches your resume and LinkedIn
[ ] Bio is one specific sentence (no buzzwords)
[ ] Contact links include LinkedIn + at least one more
[ ] Profile README exists and fits on one screen
[ ] 3-6 pinned repos, each with description + topics
[ ] Pinned repos ordered by strength (strongest first)
[ ] All pinned repo README files are complete
[ ] Live demo URLs work (no 404s)
[ ] Tutorial forks and dead repos archived or private
[ ] No secrets in any public repo history
```

## Common mistakes

| Symptom | Cause | Fix |
|---|---|---|
| Default octopus avatar | Never updated the profile picture | Upload a real photo or clean graphic |
| Bio says "Passionate ninja rockstar coder" | Buzzword reflex | Rewrite with specific technologies and focus area |
| Six pinned repos including 3 tutorial clones | Pinning to fill slots rather than to signal | Pin fewer; only pin repos with real READMEs and original work |
| Live demo URL in repo's About field returns 404 | Demo server went down or free tier expired | Fix the deployment or remove the dead link; a 404 is worse than no link |
| Contribution graph empty for 3 months during active job hunt | All work is local/private/on other platforms | Commit notes, learning repos, or small improvements to keep the graph gently active |

## Practice

**Warm-up.** Open your GitHub profile in an incognito window. Describe what a recruiter sees in the first 15 seconds. Is it accurate to your target?

<details><summary>Show solution</summary>

If your first impression doesn't match your target role (e.g., you see front-end tutorials but you're targeting backend), that's the gap to close. Note which elements are off and prioritize: bio, pinned repos, then profile README.

</details>

**Standard.** Write (or rewrite) your profile README using the template above. Limit it to one screen height.

<details><summary>Show solution</summary>

A good profile README fits without scrolling on a standard laptop screen (~30 lines of markdown). Include: greeting, one-liner, skills, featured projects with links, and current learning focus. Skip: autogenerated stats, animated graphics, walls of badges.

</details>

**Bug hunt.** Audit all your public repos. For each, note keep / archive / private / delete and why. Aim to halve the count.

<details><summary>Show solution</summary>

Create a spreadsheet with columns: repo name, last commit date, purpose, action (keep/archive/private/delete), reason. Common candidates for archive: tutorial follow-alongs, course exercises, one-file experiments. Common candidates for private: broken projects, repos with secrets in history.

</details>

**Stretch.** Rewrite repo descriptions and add topics to all pinned repos. Commit one small improvement to each.

<details><summary>Show solution</summary>

For each pinned repo: update the "About" description to one clear sentence, add 3-5 topic tags (e.g., `typescript`, `rest-api`, `postgres`, `docker`), set the homepage URL to the live demo. Commit a small improvement (update a dependency, add a test, improve error handling) to make each repo look active.

</details>

**Stretch++.** Create or polish a sixth pin that represents genuine depth — a technical writeup repo, a library you published, or a meaningful OSS contribution.

<details><summary>Show solution</summary>

Options: (1) Write a technical deep-dive as a README-only repo explaining how a system works. (2) Extract a utility from one of your projects and publish it as a standalone package. (3) Contribute a non-trivial PR to an OSS project and pin the fork with a link to the merged PR.

</details>
