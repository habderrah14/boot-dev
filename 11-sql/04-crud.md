# Chapter 04 — CRUD

> Create, Read, Update, Delete. The four things you'll do to tables for the rest of your career.

## Learning objectives

By the end of this chapter you will be able to:

- Insert rows with and without defaults, including multi-row and `INSERT … SELECT`.
- Update and delete rows safely — always with `WHERE`.
- Use `RETURNING` to get back affected rows without a follow-up `SELECT`.
- Write upserts with `ON CONFLICT`.
- Wrap multi-statement mutations in transactions for atomicity.
- Use parameterized queries to prevent SQL injection from day one.

## Prerequisites & recap

- [Tables](02-tables.md) — you can create tables with types and constraints.
- [Constraints](03-constraints.md) — you understand how the database rejects invalid data.

## The simple version

CRUD is the rhythm of every database interaction: you **C**reate rows with `INSERT`, **R**ead them with `SELECT`, **U**pdate them with `UPDATE`, and **D**elete them with `DELETE`. The critical rule is simple: `UPDATE` and `DELETE` without a `WHERE` clause affect *every row in the table*. There's no "undo" button — once you commit, the old data is gone. Transactions (`BEGIN … COMMIT`) let you group multiple changes into an all-or-nothing operation, and `RETURNING` lets you see exactly what was changed without a second query.

## Concept deep-dive

### INSERT

Single row:

```sql
INSERT INTO users (name, email) VALUES ('Ada', 'ada@example.com');
```

Multiple rows:

```sql
INSERT INTO users (name, email) VALUES
  ('Alan', 'alan@example.com'),
  ('Grace', 'grace@example.com');
```

Insert from a query (copy data between tables, generate test data):

```sql
INSERT INTO users_archive (name, email)
  SELECT name, email FROM users WHERE active = FALSE;
```

### Default and autogenerated columns

Columns with defaults can be omitted from the INSERT — the database fills them:

```sql
INSERT INTO users (name, email) VALUES ('Ada', 'a@b.c');
-- id: auto-generated, created_at: now(), active: TRUE
```

### `RETURNING`

Get the inserted, updated, or deleted row back immediately:

```sql
INSERT INTO users (name, email)
VALUES ('Ada', 'a@b.c')
RETURNING id, created_at;
```

This is a PostgreSQL feature (MySQL lacks it). It saves you a follow-up `SELECT` and eliminates race conditions: the returned row is exactly what was written, not a re-read that might reflect concurrent changes.

### UPDATE

```sql
UPDATE users SET name = 'Ada Lovelace' WHERE id = 1;

UPDATE users
SET active = FALSE
WHERE last_seen < now() - INTERVAL '1 year';

UPDATE orders
SET status = 'shipped', shipped_at = now()
WHERE id = $1
RETURNING *;
```

**Never run `UPDATE` without `WHERE` unless you mean it.** `UPDATE users SET active = FALSE;` deactivates every user in the table. There's no confirmation prompt. No undo.

### DELETE

```sql
DELETE FROM users WHERE id = 42 RETURNING *;

DELETE FROM sessions
WHERE created_at < now() - INTERVAL '30 days';
```

Same rule: always `WHERE`, unless you intentionally want to empty the table (in which case, `TRUNCATE` is faster).

### TRUNCATE

```sql
TRUNCATE TABLE sessions;
TRUNCATE TABLE sessions, session_logs CASCADE;
```

Instantly empties one or more tables. Faster than `DELETE` because it doesn't scan rows — it drops the underlying storage and recreates it. But it acquires an `ACCESS EXCLUSIVE` lock (blocks all access), resets sequences, and can't be rolled back in some engines. Use with care in production.

### Upsert (`ON CONFLICT`)

Insert if new, update if exists — in a single atomic statement:

```sql
INSERT INTO users (email, name) VALUES ('a@b.c', 'Ada')
ON CONFLICT (email) DO UPDATE
  SET name = EXCLUDED.name
RETURNING *;
```

`EXCLUDED` refers to the row that *would have been* inserted. You need a unique constraint or index on the conflict column(s). MySQL's equivalent is `ON DUPLICATE KEY UPDATE`.

### Transactions

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

If any statement fails (constraint violation, deadlock), you `ROLLBACK` and neither update is visible to other sessions. Transactions guarantee **atomicity** (all or nothing), **consistency** (constraints hold), **isolation** (concurrent transactions don't see partial results), and **durability** (committed data survives crashes).

### Parameterized queries (in application code)

Never string-concatenate user input into SQL:

```ts
// DANGEROUS — SQL injection
await client.query(`SELECT * FROM users WHERE email = '${email}'`);

// SAFE — parameterized
await client.query("SELECT * FROM users WHERE email = $1", [email]);
```

Parameterized queries send the SQL template and values separately. The engine treats `$1` as a data value, never as SQL syntax — injection is structurally impossible.

## Why these design choices

**Why `RETURNING`?** Without it, you INSERT a row, then `SELECT` it back. Between those two statements, another session could have modified the row. `RETURNING` gives you the authoritative result of the mutation in a single round-trip — correct and fast.

**Why transactions for multi-step mutations?** Imagine transferring money: debit account A, credit account B. If the debit succeeds but the credit fails (maybe B's constraint rejects the new balance), you've lost money. A transaction ensures either both happen or neither does.

**Why upsert instead of "check then insert"?** In concurrent code, two requests can both `SELECT` and find no existing row, then both `INSERT` — one fails with a duplicate violation. `ON CONFLICT` handles this atomically: the database resolves the race for you.

**When you'd pick differently.** Some write-heavy systems skip `RETURNING` to reduce network payload (if you genuinely don't need the returned data). Some architectures prefer soft-deletes (`UPDATE … SET deleted_at = now()`) over `DELETE` to maintain an audit trail. Event-sourced systems avoid `UPDATE` entirely, appending new events instead.

## Production-quality code

```sql
-- Idempotent upsert of a product, returning the final row
INSERT INTO products (sku, name, price_cents)
VALUES ($1, $2, $3)
ON CONFLICT (sku) DO UPDATE
  SET name = EXCLUDED.name,
      price_cents = EXCLUDED.price_cents
RETURNING id, sku, name, price_cents;

-- State-machine transition: only ship an order that's currently 'paid'
UPDATE orders
SET status = 'shipped', shipped_at = now()
WHERE id = $1 AND status = 'paid'
RETURNING id, status, shipped_at;

-- Atomic money transfer with balance check
BEGIN;

UPDATE accounts
SET balance = balance - $2
WHERE id = $1 AND balance >= $2;

UPDATE accounts
SET balance = balance + $2
WHERE id = $3;

COMMIT;
```

Application layer (Node.js):

```ts
import pg from "pg";
const pool = new pg.Pool({ connectionString: process.env.DATABASE_URL });

async function transferMoney(fromId: number, toId: number, amount: number) {
  const client = await pool.connect();
  try {
    await client.query("BEGIN");

    const debit = await client.query(
      "UPDATE accounts SET balance = balance - $1 WHERE id = $2 AND balance >= $1 RETURNING balance",
      [amount, fromId]
    );
    if (debit.rowCount === 0) {
      await client.query("ROLLBACK");
      throw new Error("Insufficient funds or account not found");
    }

    await client.query(
      "UPDATE accounts SET balance = balance + $1 WHERE id = $2",
      [amount, toId]
    );

    await client.query("COMMIT");
  } catch (err) {
    await client.query("ROLLBACK");
    throw err;
  } finally {
    client.release();
  }
}
```

## Security notes

- **SQL injection is the #1 database attack vector.** It's trivially preventable with parameterized queries. Never build SQL strings with concatenation, template literals, or string formatting. This applies equally to `psql` scripts, ORMs, and raw query builders.
- **`UPDATE` / `DELETE` without `WHERE` in production.** Some teams add a SQL linter or pre-commit hook that rejects unqualified mutations. Others use database roles that lack `DELETE` privileges and require a separate elevated role for data cleanup.
- **`TRUNCATE CASCADE` is nuclear.** It empties not just the target table but all tables that reference it via `CASCADE`. In production, restrict `TRUNCATE` to admin roles.

## Performance notes

- **Multi-row `INSERT` is faster than single-row loops.** Each statement has fixed overhead (parsing, planning, network round-trip). Batching 1,000 rows in one `INSERT … VALUES` is ~10x faster than 1,000 individual inserts.
- **`RETURNING` is nearly free.** The row is already in memory from the mutation — returning it adds negligible cost compared to a separate `SELECT`.
- **`TRUNCATE` vs `DELETE`.** `DELETE FROM big_table` generates a WAL entry per row and triggers per-row triggers. `TRUNCATE` resets the table file — O(1) regardless of row count. For complete table clearing, `TRUNCATE` wins by orders of magnitude.
- **Long transactions hold locks.** A `BEGIN` without a `COMMIT` for minutes blocks autovacuum and can cause lock contention. Keep transactions short.

## Common mistakes

| # | Symptom | Cause | Fix |
|---|---|---|---|
| 1 | Every row in `users` has `active = FALSE` | `UPDATE users SET active = FALSE;` — no `WHERE` clause | Always write `WHERE` first; use `EXPLAIN` or `SELECT` with same `WHERE` before mutating |
| 2 | Duplicate key violation on concurrent upsert | "Check then insert" pattern with a race condition | Use `ON CONFLICT DO UPDATE` for atomic upserts |
| 3 | Transferred money disappears: source debited, destination not credited | Two separate `UPDATE` statements without a transaction | Wrap in `BEGIN; … COMMIT;` |
| 4 | Node app vulnerable to SQL injection | String concatenation to build queries | Use parameterized queries (`$1`, `$2`) exclusively |
| 5 | Delete operation takes minutes on a large table | `DELETE FROM big_table` with triggers and FK checks | Use `TRUNCATE` for full clears; for partial deletes, batch with `WHERE id BETWEEN … LIMIT …` |

## Practice

**Warm-up.** Insert two users with `RETURNING *`. Verify both rows are returned.

**Standard.** Create a `products` table with a `sku TEXT UNIQUE` column. Write an upsert that inserts a product or updates its name and price if the SKU already exists. Test it twice with the same SKU but different values.

**Bug hunt.** A teammate ran `UPDATE users SET active = false;` in production (no `WHERE`). All 50,000 users are now inactive. What's your mitigation plan? How would you prevent this in the future?

**Stretch.** Create an `accounts` table with a `balance` column and a `CHECK (balance >= 0)`. Write a transaction that transfers money between two accounts. Verify that overdrawing is rejected by the constraint and rolls back both updates.

**Stretch++.** Write an idempotent insert: if the row already exists (by email), return the existing row *without* updating it. (Hint: `ON CONFLICT … DO NOTHING` + a fallback `SELECT`.)

## Chapter summary

- CRUD = `INSERT` / `SELECT` / `UPDATE` / `DELETE` — the four operations you'll use on every table.
- Always use `WHERE` on mutations unless you intentionally want to affect all rows.
- Use `RETURNING` to avoid follow-up queries, `ON CONFLICT` for atomic upserts, and transactions for multi-step atomicity.
- Parameterized queries (`$1`, `$2`) prevent SQL injection — this is non-negotiable.
